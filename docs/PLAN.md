# LDLAWQ 改进计划 v1.0（2026-06-11）

> 核心目标：以**固定法条库 + 真实案例库**为唯一知识来源，让 AI 精准回答企业 HR 的
> 劳动法 / 用工问题，达到可替代律师日常咨询答疑的水平；疑难个案仍转律师（兜底 +
> 律协渠道动力）。界面同时支持移动端与 PC。
>
> 配套文档：[PRD.md](PRD.md)（产品全貌）· [README.md](../README.md)（当前状态）
> **执行层**：本计划的全部任务已拆解为 Opus 4.8 可独立执行的任务卡，见 [EXECUTION.md](EXECUTION.md)
> （进度以其 §0 状态总览为唯一真相源，本文不再重复维护进度）。

"取代律师"的可达边界（写进产品话术，也写进路由逻辑）：

- **AI 接管**：高频规则明确的咨询（约占 HR 日常提问的 70–80%）——补偿计算、
  合同签订时限、年假/加班口径、流程合规性、文书怎么写。
- **必须转律师**：个案争议研判、证据策略、仲裁/诉讼代理、出具法律意见书。
  2008 年前入职分段计算等已知复杂口径，直接强制转律师路由（而非仅文字提示）。

## 3. P1 知识层扩容（W1–W4）——本计划的重心

### 3.1 法条库：从 11 条到"全国骨架完整 + 沪苏地方落地"

**全国性法规（M0 必备，约 400–600 条）**：

| 类别 | 法规 |
|---|---|
| 法律 | 劳动法、劳动合同法、劳动争议调解仲裁法、社会保险法、工会法（涉解除程序条款） |
| 行政法规 | 劳动合同法实施条例、职工带薪年休假条例、工伤保险条例、女职工劳动保护特别规定 |
| 司法解释 | 最高法劳动争议司法解释（一）、新司法解释（二，2025-09-01 施行）——口径变化大，必须入库 |
| 部门规章 | 带薪年休假实施办法、工资支付暂行规定、劳社部发〔2008〕3 号（21.75 口径）等 |

**地方层（M0 先沪苏两地，这是产品护城河的第一块砖）**：
上海/江苏的工资支付办法、高院与人社的审理口径纪要、特殊工时审批口径等。
每条打 `region_id` 标签，检索时按"全国 → 省 → 市"链路过滤（机制已有）。

**核验流程（决定 verified=1 的生产门槛）**：
1. 运营从官方源（flk.npc.gov.cn / 各地人社官网）逐字粘贴，登记 `source_url`
2. 第二人比对原文 → `verified=1, verified_by, verified_at`
3. 律师抽检每法规 ≥20% 条文
4. 构建脚本现有的"未核验计数"升级为：**M0 demo 允许 ⚠ 展示，M1 试点门槛 = 0**

需要新建：`src/ingest_law.py`（半自动入库：贴全文 → 自动切条 → 人工确认）。
纯手工录入 400 条不现实，切条器是本阶段最值得写的工具。

### 3.2 案例库 v1（用户核心需求，从 0 到 1）

**定位**：案例不直接当答案，干三件事——
① 相似案情检索增强（RAG 上下文里"法条 + 裁判要旨"并排）；
② 答案中的"裁判倾向提示"（如"上海法院近年对调岗合理性的把握……"，必须挂案号）；
③ 评测集的金标准来源。

**获取渠道（PRD 既定三渠道，按可行性排序）**：

| 渠道 | M0 可得量 | 说明 |
|---|---|---|
| 官方发布 | 200–500 件 | 最高法指导性案例/典型案例、人社部+最高法联合发布的劳动争议典型案例、沪苏法院年度劳动争议十大案例——公开、权威、自带裁判要旨，**首选** |
| 律师脱敏 | 30–50 件 | 合作律师提供，脱敏后入库（`anonymized=1`），律协渠道的天然贡献 |
| 商业授权 | M1 再谈 | 威科先行/北大法宝 API，按预算定 |
| ~~爬裁判文书网~~ | 0 | 不做，合规红线（既定决策） |

**结构化入库**（`case_record` 表已建好）：案号、法院、地区、争议焦点标签
（`dispute_tag`）、案情摘要、**裁判要旨（gist，检索与生成的主字段）**、结果、
关联法条（`case_citation`，与法条库硬关联——案例引用同样过存在性校验）。

**M0 目标：100 件高质量案例**（官方为主），覆盖 TOP 争议类型：违法解除、
未签合同二倍工资、调岗调薪、加班费、竞业限制、三期女职工、年休假。

### 3.3 词条与参数

- 词条从 2 条扩到 **TOP 30**（按律师反馈的 HR 高频问题清单），每条走
  `in_review → published` 律师审核流（`review_log` 表已建好，接上）
- 地区参数补齐沪苏**市级**粒度：社平工资（封顶基数法定就是市级口径——现省级
  参数是口径错误，必须改）、最低工资、一次性工亡补助基数等，全部带生效期间

## 4. P2 检索与回答精度（W2–W6）

### 4.1 检索升级：bigram → 混合检索

```
查询 ──┬─ FTS5（中文分词 tokenizer）──┐
       │                              ├─ 加权融合 + 地区过滤 → 候选 8–12 条
       └─ sqlite-vec 向量（bge-m3）──┘
                  ↓
        法条 / 案例 双索引分别检索，分别进入 RAG 上下文
```

- 嵌入模型：bge-m3（本地跑得动、中文法律文本表现好）；嵌入在**构建期**算好
  存入 knowledge.db，线上零推理依赖
- 保留"检索置信度不足 → 不送生成"闸门，阈值用评测集标定（替掉现在拍脑袋的 score<2）

### 4.2 回答管线升级

- **多轮对话**（关键缺失）：现在 clarify 后用户补充信息是全新请求，上下文丢失。
  接通 `qa_session`：同 session 内要素累积合并，补一个要素就能继续算
- RAG 上下文 = 法条原文 + 相似案例裁判要旨（标明案号），prompt 要求区分
  "法定规则"与"裁判倾向"两个层次作答
- 校验器扩展：案例引用也过存在性校验；答案涉及地方差异时强制带地区适用性声明
- 年假计算口径修正：headline 改为"额外应补 200%"为主、300% 总额并列；
  当年新入职按"本单位已过日历天数"折算；hire<2008 强制转律师路由
- 复杂问题可升级 deepseek-reasoner（要素齐全但推理链长的场景），按评测收益定

### 4.3 评测体系（"非常精准"的唯一证明方式）

- **评测集 100 题（M0）→ 300 题（M1）**：律师出题，每题带金标准结论 +
  金标准引用 + 地区（`eval_item` 表已建好）
- `src/run_eval.py` 自动跑分，四个硬指标：

| 指标 | M0 门槛 | M1 门槛 |
|---|---|---|
| 编造引用率 | **= 0**（PRD 既定红线） | = 0 |
| 结论正确率（律师判分） | ≥ 85% | ≥ 92% |
| 拒答恰当率（该拒的拒、不该拒的不拒） | ≥ 80% | ≥ 90% |
| 引用完整率（金标准引用命中） | ≥ 75% | ≥ 85% |

- **纪律：知识库、prompt、检索参数任何变更 → 必须跑评测，分数回退不合入**
  （结果写 `eval_run` 表，git 提交信息带分数）

## 5. P3 界面双端（W4–W7）

现状：单文件响应式 SPA，375px 移动端已验证——底子是对的，**继续单代码库
响应式，不做独立 App**。改进点：

- **PC 宽屏（≥1100px）双栏布局**：左栏对话流，右栏"依据面板"（命中法条原文 +
  相似案例常驻展示，律师工作台式体验——这是 PC 端相对手机的核心增益）
- **移动端**：现有四页签保留；问答页增加多轮要素补全的快捷交互
  （clarify 项变成可点选/快速填写的表单 chip，而不是让用户打字）
- 新增**案例页签**：按争议类型/地区筛选浏览，详情页展示裁判要旨 + 关联法条
- 新增**文书模板页签**（M1）：分类下载 + 填写指引 + 风险提示（表结构已建好）
- 回答 UI 强化信任感：法条/案例引用逐条可展开原文（已有）+ 核验状态 +

- 入口规划：M1 加微信 H5 适配（HR 的真实使用场景在微信里），小程序壳 M2 再议

## 8. 风险与对策

| 风险 | 等级 | 对策 |
|---|---|---|
| 法条/案例录入与核验人力不足（最可能拖垮 M0） | 高 | 切条工具半自动化；核验众包给律协（飞轮本来就是这么设计的）；M0 范围死守沪苏不扩 |
| 案例官方渠道量不够 | 中 | 官方典型案例 + 律师脱敏双轨；不够就降级为"法条为主、案例为辅"，精度指标不变 |
| 评测分数达不到门槛 | 中 | 这恰恰是评测的价值——按失分类型回灌：检索问题补语料、口径问题补词条、推理问题换 reasoner |
| 备案结论晚于 M1 | 中 | M1 按"邀请制内测"设计，绕开公开服务认定，同时推进备案 |
| "取代律师"话术引发律协渠道反弹 | 低 | 对外口径统一为"AI 处理高频咨询、律师专注疑难个案与案源"，转介数据透明给律协 |
